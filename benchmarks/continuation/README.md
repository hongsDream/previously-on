# GPT-5.5 / GPT-5.6 continuation benchmark

This benchmark measures whether a coding workflow survives repeated context compaction better in the same Codex task or through a source-model-generated native handoff to a fresh task. It is separate from release compatibility evidence and must never be used to rerun the prior 60 live compatibility workflows.

The benchmark is bound to Regression Contracts merge/main SHA `7aba2ba71a40b713be54d3540386fa4195026354` and fixture-set SHA-256 `4f6ae416871a9a502377cad82396053883ab0aebaa479c016baad3d3f7363b4a`.

## Fixed base design

- Models: `gpt-5.5` and `gpt-5.6-sol`; reasoning `high`; fast service tier off.
- Scenarios: four offline synthetic coding repositories and four workflows based on the verified PreviouslyOn merge.
- Strategies: same task and a byte-preserved native handoff generated by the source model.
- Compaction checkpoints: `0,2,4,6,8,10,12,14,16`; three repetitions.
- Base matrix: `2 × 8 × 2 × 9 × 3 = 864` measured arms and 432 paired source snapshots.
- Product strategy: native handoff plus a verified `resume_task` Context Pack and only relevant active Regression Contracts. It is scheduled only after a model-specific, CI-supported boundary and odd-checkpoint refinement.

Every arm receives an isolated Git workspace at the fixture snapshot. Source threads are read-only. Challenge threads may write only inside their arm workspace. The full goal, invariants, authorized files, stale facts, exact test argv, and result identifiers are supplied only in the initial source state; the final challenge is deliberately generic so it cannot refill lost continuation state. The harness independently inspects the diff, reruns the fixture argv test, verifies that the expected number of tests actually executed, and runs a SHA-pinned hidden oracle from outside the writable repository. The oracle does not exist anywhere under the arm root until the challenge turn and declared test have ended, and every workspace reset removes a previously materialized copy before another model turn. PreviouslyOn workspaces receive only a deterministic ignored `ui/dist/index.html` compile input when the pinned source embeds that directory; it is not part of the scored diff. Model claims and model-editable tests are not accepted as sole success evidence. Raw source prompts, challenge responses, transcripts, and tool output are not retained in result records. The source-model native handoff text is retained only in the local, ignored, secret-scanned control ledger until its fresh-task arm completes so an interrupted run can deliver exactly the same bytes without another paid turn; versioned and derived outputs retain only its hash and bounded metrics.

The campaign lock hashes the executable harness, schemas, hidden oracles, and every synthetic repository-template byte. A template edit therefore creates a different campaign instead of mixing different code snapshots under one fixture/campaign identity.

The runner fsyncs a content-hashed paid-stage checkpoint immediately after each source initial, worklog, and native-handoff turn, then freezes a source snapshot after every completed compaction—not only the even base checkpoints. Source checkpoints form a campaign/fixture/source/sequence hash chain. Compaction intent and completion records are also hash-bound, and resume verifies the official `thread/read` snapshot and observable compaction state before it can continue; a crash after compaction but before the next source checkpoint is recovered without issuing a second compaction. An interruption at any post-turn usage guard resumes the persisted App Server thread/stage instead of paying for that turn again. This also lets later odd-checkpoint refinement fork the exact historical source state. Every fixture contains 20 fixed worklog turns, so the `18,20` extension is executable without changing fixture bytes.

Model identity is observed separately for every paid source, worklog, native-handoff, and challenge turn, preferring a provider-exposed snapshot and otherwise retaining the exact verified/resolved catalog model ID. Conflicting identities or an explicit failed verification fail the arm closed; if any paid stage lacks either form, the arm may remain measured but that model cannot produce a rollover recommendation.

## Files

- `manifest.v1.json`, `schemas/`, and `fixtures/`: versioned experiment contract.
- `repositories/`: deterministic, offline synthetic repository baselines.
- `oracles/`: versioned semantic oracles copied read-only outside each challenge repository and bound by fixture SHA-256.
- `runner.mjs`: App Server runner, checkpoint/resume logic, 80% usage guard, and campaign lock.
- `src/workspace.mjs`: isolated workspace creation, diff oracle, argv test capture, and redaction.
- `src/scorer.mjs`, `src/statistics.mjs`, `src/summary.mjs`: deterministic scoring, seeded 10,000-sample paired scenario-cluster bootstrap, same-task/native-handoff comparisons, boundary and product gates.
- `src/product-arm.mjs`: public `resume_task` MCP plus `previously contracts check --execute --json` composition. It preserves the MCP trust envelope and removes irrelevant/superseded contracts.
- `failure-taxonomy.v1.json`: serious, state-recall, infrastructure, privacy, and product-verification failure classes.
- `results/`: versioned dry-run/checkpoint/recommendation artifacts. Live JSONL, campaign locks, and workspaces remain local because they can contain repository metadata.

## Official provider provenance gate

Live execution accepts only a directly executable binary that reports a `codex-cli` semantic version and whose App Server `initialize` response identifies a Codex user agent. The binary path, bytes SHA-256, version, initialize user agent, and macOS `codesign --verify --strict` observation are bound into both calibration and measured campaign locks. A missing or invalid local signature is recorded rather than promoted into an unsupported product requirement; model execution still uses only the documented App Server JSONL interface: initialization, model catalog, thread start/resume/fork/read, turn start, explicit compaction, account rate-limit reads, and account usage reads.

The runner fails closed if the version, initialization identity, required model catalog entries, official rate-limit coverage, calibration bindings, or any campaign hash differs. Metrics the App Server does not expose are recorded as unavailable rather than inferred.

## Summary, extension, refinement, and product arm

Generate the derived recommendation after any measured window:

```bash
node benchmarks/continuation/summarize.mjs \
  --results benchmarks/continuation/results/measured-integrity-20260715/results.v1.jsonl \
  --json benchmarks/continuation/results/threshold-recommendation.v1.json \
  --markdown benchmarks/continuation/results/threshold-recommendation.v1.md
```

For every model and checkpoint, the JSON and Markdown summaries pair identical scenario/repetition/fixture arms and report `native_handoff - same_task` accuracy and serious-error percentage-point differences, completion and end-to-end median latency percentage changes, and the four goal/changed-files/test-status/next-step recall differences. Every metric includes a seeded scenario-cluster paired 95% bootstrap interval. Incomplete condition, latency, or recall coverage is recorded as `unavailable`; it is never silently calculated from a smaller subset.

If no supported boundary exists through checkpoint 16, the summary requests checkpoints 18 and 20. If an even-grid boundary appears, it requests adjacent odd checkpoints before the product gate. Generate and execute that model-specific schedule:

```bash
node benchmarks/continuation/schedule.mjs \
  --mode adaptive \
  --summary benchmarks/continuation/results/threshold-recommendation.v1.json \
  --results benchmarks/continuation/results/measured-integrity-20260715/results.v1.jsonl \
  --output benchmarks/continuation/results/adaptive-schedule.v1.json

node benchmarks/continuation/runner.mjs \
  --phase measured \
  --app-server-bin /Applications/ChatGPT.app/Contents/Resources/codex \
  --results benchmarks/continuation/results/measured-integrity-20260715/results.v1.jsonl \
  --control benchmarks/continuation/results/measured-integrity-20260715/control.v1.jsonl \
  --campaign-lock benchmarks/continuation/results/measured-integrity-20260715/campaign-lock.v1.json \
  --calibration-results benchmarks/continuation/results/calibration-integrity-20260715/results.v1.jsonl \
  --calibration-control benchmarks/continuation/results/calibration-integrity-20260715/control.v1.jsonl \
  --calibration-lock benchmarks/continuation/results/calibration-integrity-20260715/campaign-lock.v1.json \
  --schedule benchmarks/continuation/results/adaptive-schedule.v1.json \
  --resume
```

Schedule generation recomputes its summary from the current append-only same-task/native-handoff records, binds their canonical digest and campaign lock, and rejects a supplied summary with different bytes. Before executing any adaptive or product arm, the runner repeats that derivation and requires the complete arm matrix and boundary proof to match exactly. After refinement leaves no pending checkpoints, generate the conditional product schedule with `--mode product` and the same `--results` argument. Each product arm must have an exact arm-key entry in a product-context map. Create those entries only after the source App Server task has been explicitly imported into PreviouslyOn; use the public product seam:

```bash
node benchmarks/continuation/materialize-product.mjs \
  --previously-bin /absolute/path/to/previously \
  --data-dir /absolute/path/to/isolated-data \
  --repository /absolute/path/to/bound-fixture-repository \
  --repository-id REPOSITORY_ID \
  --task-id SOURCE_TASK_ID \
  --fixture-sha FIXTURE_SHA256 \
  --fixture benchmarks/continuation/fixtures/SCENARIO.json \
  --base BASE_SHA \
  --head HEAD_SHA \
  --source-key MODEL/SCENARIO/REPETITION \
  --source-compaction COMPACTION_CHECKPOINT \
  --source-thread-id SOURCE_SNAPSHOT_THREAD_ID \
  --source-snapshot-sha SOURCE_SNAPSHOT_SHA256 \
  --output /absolute/path/to/product-context.json
```

`--base` is the fixture's logical `repositorySnapshot.baseSha` (including a synthetic fixture's versioned logical SHA); `--head` is the actual commit checked out in the isolated arm repository. Copy `sourceKey`, `compaction`, `snapshotThreadId`, and `sourceSnapshotSha256` from the exact `source_checkpoint` control record frozen for that arm. The snapshot digest is calculated from the official `thread/read(includeTurns=true)` response, not fixture metadata. A product artifact is valid only for that model/scenario/repetition checkpoint and must not be reused across repetitions or compaction checkpoints. Contract selection runs in a throwaway clone with a deterministic impact probe, so a synthetic logical base and a `base == HEAD` PreviouslyOn fixture can both produce a real relevant-contract evaluation without mutating the arm repository. The product map is a JSON object from exact arm key to the corresponding product-context path. Before product materialization or any fresh challenge, the runner requires the completed native-handoff arm for the same model/scenario/repetition/checkpoint/fixture and the same frozen source snapshot. It reuses that arm's retained source-model handoff bytes instead of generating a second handoff, and binds the native terminal-record SHA plus paid-stage checkpoint SHA into the product result. Missing or mismatched source snapshot, handoff SHA, delivered SHA, checkpoint SHA, or byte length fails closed. The summary repeats this pairwise provenance check before evaluating the product gate; invalid pairs can only yield `no_auto_rollover`. Materialization also calls public `get_task_timeline` and requires the imported PreviouslyOn task to contain that exact App Server source thread with the exact compaction count. Product execution fails closed if this provenance, its Context Pack trust envelope, repository/task identity, hash, fixture/base/head binding, readiness, required-test evidence, or relevant active contract set is invalid. The real `resume_task` call becomes available only after the measured source task is imported into PreviouslyOn.

## Decision rules

A degradation boundary requires the same metric to cross its point threshold and adverse-side 95% bootstrap CI at two consecutive evaluated checkpoints: success at least 10 percentage points lower than checkpoint 0, serious errors at least 10 points higher, or the same completion/E2E median at least 25% slower. Single spikes and alternating metrics do not count.

The product arm passes only when all 48 unique scenario/repetition/checkpoint pairs and their timing metrics are present, its accuracy lower CI is greater than `-5pp`, its serious-error point estimate and upper CI do not increase, and it improves either accuracy by at least `+5pp` or median time by at least 15%. Product end-to-end time includes the hash-bound duration of the real `resume_task`/timeline/Regression Contract materialization plus artifact loading; completion time remains the challenge model's completion latency. A recommendation additionally requires a complete base matrix, completed refinement, one stable observed model identity across every paid stage, complete result-record/campaign/fixture/manifest bindings, and complete serious-error coverage. Otherwise the emitted action is `no_auto_rollover`; no result is generalized to an untested model or snapshot.
